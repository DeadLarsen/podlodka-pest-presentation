### Слайд 3 — Почему важны тесты в эпоху AI-кодинга?
**[01:15 — 03:15]**



AI не понимает ваш бизнес-контекст. Он угадывает.
Не против  погаллюцинировать — уверенно пишет код, который выглядит правильно, но работает не так как мы того хотим. 
Он крашит архитектуру — кладёт файлы на свое усмотрение, создаёт циклические зависимости. 
И код ревью не масштабируется — человек физически не успевает проверять весь AI-сгенерированный код.
 
Что остаётся? Тесты. Тесты — единственный автоматический механизм, который скажет: «нет,большой брат, тут сломалось». Из «хорошей практики» тесты превращаются в критическую инфраструктуру. Если раньше можно было обойтись без них — теперь нельзя.

### Слайд 6 — Пирамида vs  Кубок
**[07:00 — 09:00]**

Прежде чем писать тесты — нужна стратегия. Два подхода.

Классическая пирамида : внизу много юнит-тестов — быстрые, дешёвые. В середине — интеграционные. Наверху — мало E2E, дорогие и медленные. Правило: чем выше уровень, тем меньше тестов.

### Слайд 6 — Пирамида vs Кубок
Кубок — модель: внизу статический анализ — линтеры, типы. Потом юниты. Фокус — на интеграционных тестах, потому что они дают максимальный ROI ( (Return on Investment) — это показатель рентабельности или окупаемости инвестиций, который демонстрирует прибыльность или убыточность бизнеса/проекта). И наверху — минимум E2E.



Что интересно: Pest задевает все уровни обеих моделей тестирования. Static-уровень — это arch-тесты. Unitы. Интеграционные. E2E — browser testing. И поверх всего — mutation testing как метрика качества. PHPUnit покрывает только unit и integration. Для остального нужны отдельные инструменты.

### Слайд 7 — TaskFlow: стратегия
**[09:00 — 10:00]**

Для нашего TaskFlow предлагаю взять модифицированный кубок. Допустим творческую вольность и распределим тесты по кубку таким образом

Почему кубок, а не пирамида — для TaskFlow
«это trade-off, который мы выбрали осознанно»
Пирамида и кубок — не про то, что одна модель лучше. Это про то, где у вашей команды главный источник багов. Для TaskFlow мы выбрали кубок, потому что:

80% наших багов в проде за последний год были в интеграциях модулей, а не в чистой логике
А Наши юнит-тесты с моками ловили <10% регрессий
AI-инструменты хорошо генерируют happy-path юниты, но плохо интеграционные сценарии — значит, именно интеграционные пишет человек осознанно (ну как пишет, скорее направляет ИИ) 

Если у вас критичная финансовая логика с сотнями правил — пирамида правильнее. Если у вас приложение, собранное из модулей и внешних сервисов — кубок даёт больше уверенности за те же тесто-часы».

Но это отдельная тема для холивара

### Слайд 8 — Уровень 0: Arch-тесты
**[10:00 — 12:00]**

И так

Первая линия обороны — без единого assert. Только правила.

Arch-тесты в Pest — это автоматические проверки правил архитектуры вашего кода, которые запускаются как обычные тесты: «модели должны лежать в App\Models», «контроллеры не должны использовать dd()», «сервисы не зависят от HTTP-слоя». Они не выполняют код — они анализируют его структуру и падают, если кто-то (человек или AI) нарушил договорённости команды.

Под капотом nikic/php-parser

### Слайд 8 — Уровень 0: Arch-тесты
**[10:00 — 12:00]**

Рассмотрим поближе

Запрет дебаг-кода: `arch('no debugging')->expect(['dd', 'dump', 'var_dump', 'ray'])->not->toBeUsed()`. Всё. Один тест — и dd() никогда не попадёт в прод.

Модели только в Models-неймспейсе. 
 
Сервисы не зависят от контроллеров. 

Готовые пресеты — `arch()->preset()->php()` для базовых правил, `arch()->preset()->security()` для безопасности.

Подчеркну: это уникальная фича Pest. В PHPUnit arch-тестов не существует. Для такого нужен отдельный инструмент вроде Deptrac. В Pest — одна строка.

И это идеальные guardrails для AI. AI нагенерировал код, нарушил зависимости — arch-тест упал. Автоматически, без ревью.

### Слайд 12 — Mutation Testing
**[17:00 — 18:30]**

Поверх всех уровней — mutation testing. Pest берёт ваш код, мутирует: меняет умножение на сложение, минус на плюс, убирает return. Запускает тесты. Если тесты прошли — значит, они ничего не проверяют.


Mutation Score 100% — это отличная метрика качества. Покрытие кода может ввести в заблуждение, если строка  «покрыта» (простите за тавтологию), но тест при этом не проверяет её результат. Мутационное тестирование выявляет именно такие «пустые» тесты

За это отвечает pest-plugin-mutate с собственным мутатором под капотом

### Слайд 13 — Скорость CI
**[18:30 — 19:30]**

Практика: CI для TaskFlow. `--parallel` запускает тесты на всех ядрах. `--shard=1/3` разбивает на три части. В GitHub Actions — матричная стратегия: три джобы по 40 секунд вместо одной на 12 минут. 
По факту это директива GitHub Actions, которая говорит: «возьми этот джоб и создай из него 3 копии, каждая с разным значением matrix.shard». То есть вместо одной джобы tests вы получите три: tests (1), tests (2), tests (3). Все они запустятся параллельно на разных раннерах

### Слайд 16 — Named arguments и пустые datasets
**[22:00 — 23:00]**

Два бага с datasets. Первый — issue #1344: если используете именованные аргументы PHP 8, Pest их теряет при генерации класса-обёртки для PHPUnit. Получите deprecation warning. Workaround — позиционные аргументы.

Второй — issue #1384: если dataset-функция возвращает пустой массив, вместо понятной ошибки — крэш с невнятным сообщением. Всегда проверяйте, что генератор возвращает хотя бы один элемент.


Issue #1483 — коварный баг. После апгрейда на v4 тесты в parallel-режиме показывают все как passed, но exit code — единица. CI фейлит при зелёных тестах. Те же тесты последовательно — всё ок.

Issue #1549 — arch-тесты игнорируют exclude из phpunit.xml. Если у вас большие сидеры — 800 тысяч строк — arch-тест будет сканировать и их. 80 секунд вместо шести. Workaround — `->ignoring()` в самом тесте.


Несколько связанных issues. Навигация в браузерных тестах может вызвать silent crash — процесс умирает без ошибки. Субдомены не работают — хост захардкожен. Каждый visit() создаёт новый контекст — cookies не разделяются.

Code coverage в v4 на 20% медленнее, чем в v3 — ирония для фреймворка, который ускорил type coverage. 

Browser Testing — мощная фича, но молодая. Для новых проектов — отличный старт. Для миграции с Dusk в продакшене — следите за issues.

Ну хватит о плохом

### Слайд 21 — Почему Pest хорош для AI
**[27:30 — 29:00]**

Пять аспектов  Pest в AI-воркфлоу.

Guardrails (Защитные ограждения) — arch-тесты ловят нарушения архитектуры AI-кодом до ревью. 
Человекочитаемость — `it('validates email')` мгновенно проверяется глазами. 
Генерация — лаконичный синтаксис, меньше токенов, AI генерирует точнее. 
Качество — mutation testing показывает, что AI-тесты реально проверяют, а не создают иллюзию. 
И скорость — AI генерирует 80% тестов за минуты, а mutate находит оставшиеся 20%.

### Слайд 5 — Pest vs PHPUnit
**[04:30 — 07:00]**
и раз уж зашла речь про сравнение

Давайте сравним в лоб. Таблица на экране.

Синтаксис: PHPUnit — классы, методы, `$this->assert`. Pest — функции `it()`, `test()`, `expect()`. Меньше кода, яснее намерение.

Expectation API: у PHPUnit нет цепочек ожиданий — только assert-методы. У Pest — fluent (текущая апи (Главная идея: сделать код максимально читаемым, чтобы цепочка вызовов методов выглядела как естественное предложение на человеческом языке (похоже на внутренний DSL — domain-specific language).)) API с цепочками.

Arch-тесты: у PHPUnit их нет. У Pest — встроенные, с пресетами для PHP, Laravel, security.

Mutation Testing: в PHPUnit — нужен отдельный фреймворк Infection  с конфигурацией. В Pest — встроенный `--mutate`, без настройки.

Browser Testing: PHPUnit — так же отдельное решение. Pest v4 — встроенный плагин на Playwright.

Datasets: PHPUnit — `@dataProvider` и отдельный метод. Pest — `->with([])` прямо рядом с тестом.

Параллельность, watch mode, красивый вывод — у PHPUnit нет, у Pest есть.

И ключевое: Pest — это надстройка над PHPUnit. Все assert-методы PHPUnit продолжают работать. Это не «или-или» — это «и».

### Слайд 25 — Best Practices
**[35:00 — 36:30]**


Best Practices: указывайте expect, не assert. 
Требуйте datasets.
Проверяйте через mutate. 
Пишите arch-тесты до AI-генерации кода — это guardrails, которые ловят нарушения сразу.
Добавляйте covers для mutation testing.

Не делайте: не доверяйте AI-тестам без mutate — AI отлично генерирует happy path, но пропускает edge cases. Не генерируйте без контекста. Не используйте assert — AI путается между стилями. И не копируйте без понимания — если не понимаете тест, он бесполезен.

### Слайд 22 — AI-скиллы для Pest
**[29:00 — 31:30]**

Как правильно промтить AI для Pest-тестов. У меня есть шаблон промта, который даёт стабильно хорошие результаты.

Ключевые моменты: указывайте стиль — expect API, не assert. Формат — describe/it, BDD. Требуйте datasets для вариаций. Покрытие — happy path плюс edge cases плюс negative. И обязательно — `->covers()` для mutation testing.

Посмотрите на результат: describe-блок для TaskService::complete. Три теста: успешное завершение, отклонение уже завершённой задачи, отправка уведомления. Каждый с `->covers()`. Это рабочий код, который можно сразу запустить.

Обратите внимание — AI сгенерировал happy path и один negative case. Mutation testing покажет, если не хватает проверки граничных условий. Вот почему `--mutate` — обязательный шаг после AI-генерации.
